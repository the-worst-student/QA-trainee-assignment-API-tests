## Общие сведения о тестах

- Все тесты рассчитаны на общий стенд.
- Для каждого запуска используются уникальные `sellerID` и уникальные `name`.
- Тесты не зависят от фиксированного количества объявлений в системе.
- При проверке списков объявлений используется поиск созданного объекта по `id`.

---

## Покрываемые endpoint'ы

- `POST /api/1/item`
- `GET /api/1/item/{id}`
- `GET /api/1/{sellerID}/item`
- `GET /api/1/statistic/{id}`

---

## Используемые техники тест-дизайна

- Эквивалентное разбиение
- Error guessing
- Проверка повторных запросов
- End-to-end сценарий
- Контрактные проверки
- Нефункциональные проверки времени ответа и устойчивости

---

## Позитивные сценарии

### TC_API_001 — Создание объявления с валидными данными

**Тип:** Позитивный, функциональный  
**Приоритет:** High

**Шаги:**
1. Отправить `POST /api/1/item` с валидным JSON.
2. Проверить успешный ответ.
3. Извлечь `id` созданного объявления из фактического ответа.
4. Выполнить `GET /api/1/item/{id}`.
5. Проверить поля объявления.

**Ожидаемый результат:**
- `POST` возвращает `200`.
- Объявление успешно создаётся.
- `GET /api/1/item/{id}` возвращает созданный объект.
- Поля `sellerId`, `name`, `price`, `statistics`, `createdAt` корректны.

---

### TC_API_002 — Получение созданного объявления по id

**Тип:** Позитивный, функциональный  
**Приоритет:** High

**Шаги:**
1. Создать объявление.
2. Извлечь `id` из create-ответа.
3. Выполнить `GET /api/1/item/{id}`.

**Ожидаемый результат:**
- Ответ успешный.
- Возвращается объект с нужным `id`.
- Поля объекта совпадают с отправленными значениями.

## Важное замечание по контракту

Во время фактического запуска выявлено расхождение между примером Postman collection и реальным поведением `POST /api/1/item`.

### По примерам Postman collection ожидалось:
JSON с полями:
- `id`
- `sellerId`
- `name`
- `price`
- `statistics`
- `createdAt`
