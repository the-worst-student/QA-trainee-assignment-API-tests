## Фактический ответ `POST /api/1/item` не соответствует примеру из Postman collection

**Приоритет:** P1  
**Серьёзность:** Major  
**Тип дефекта:** Несоответствие контракта API / документации / Postman collection

### Окружение
- Host: `https://qa-internship.avito.com`
- Endpoint: `POST /api/1/item`
- Версия API: `v1`
- Клиент: `pytest + requests`

---

## Краткое описание

Для ручки создания объявления фактический ответ сервера не соответствует примеру, указанному в Postman collection.

По Postman collection после успешного создания объявления ожидается полноценный JSON-объект объявления с полями:
- `id`
- `sellerId`
- `name`
- `price`
- `statistics`
- `createdAt`

Фактически API возвращает только поле `status`, внутри которого идентификатор объявления зашит в текстовую строку.

Из-за этого контракт `POST /api/1/item` в реальном стенде отличается от задокументированного в collection.

---

## Что указано в Postman collection

В Postman collection для успешного ответа `POST /api/1/item` приведён пример такого вида:

```json
{
  "id": "<string>",
  "sellerId": "<integer>",
  "name": "<string>",
  "price": "<integer>",
  "statistics": {
    "likes": "<integer>",
    "viewCount": "<integer>",
    "contacts": "<integer>"
  },
  "createdAt": "<string>"
}
```

Фактический ответ сервера имеет вид:

````{
  "status": "Сохранили объявление - 08312027-614c-478e-a4ff-97b1477ec461"
}
````
То есть: отдельного поля id в JSON нет;
полей ``sellerId, name, price, statistics, createdAt`` тоже нет;
идентификатор объявления содержится только внутри текстового поля status.

Фактический контракт не совпадает с документированным.
Клиент, написанный по collection, ожидает одно поведение, а получает другое.
Ответ create-ручки неудобен для автоматизации и интеграций.
Полезные поля объекта отсутствуют в create-ответе.
Клиент не может сразу провалидировать созданную сущность без дополнительного GET.

Из-за этого дефекта: автотесты, написанные строго по Postman collection, падают;
create-ответ нельзя использовать как полноценный источник данных о созданном объекте;
id нельзя взять как ``response.json()["id"];``
приходится внедрять дополнительную логику извлечения id из строки status;
после POST требуется дополнительный GET ``/api/1/item/{id}``, чтобы получить сам объект и провалидировать его поля.
